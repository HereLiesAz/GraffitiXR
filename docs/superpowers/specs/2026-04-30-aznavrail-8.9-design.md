# AzNavRail 8.9 Migration + Help/Tutorial Compliance + Content Audit

**Status:** Spec 1 of 3 (sequential). Predecessors: none. Successors: co-op realtime sync; wearables glasses-as-camera.

**Date:** 2026-04-30
**Owner:** Az
**Library version:** com.github.HereLiesAz:AzNavRail 8.8 → 8.9

---

## 1. Goal

Bring GraffitiXR's AzNavRail integration into structural and content compliance with AzNavRail 8.9. "Compliance" here is the user-defined "Option C": API alignment with the 8.9 documented surface **plus** an audit pass over every help string and tutorial card for length, voice, and accuracy. Eliminate the recurring `IllegalArgumentException` class of bugs by structural means, not patches.

## 2. Non-goals

- New help/tutorial coverage for features that don't currently have any. Existing coverage gets audited; gaps are noted but not filled.
- Localization correctness. English content is reviewed; localized strings are out of scope.
- Visual snapshot/screenshot testing of the overlay.
- Anything in `collab/` (Spec 2 territory).
- Anything in `core/common/.../wearable/` beyond what's needed to keep the rail item present (Spec 3 territory).

## 3. Background

### 3.1 Current state

GraffitiXR uses AzNavRail 8.8. The integration is more 8.9-aligned than it appears at first read. `MainActivity.kt:434-462` already wraps the UI in `AzHostActivityLayout` with the documented configuration blocks:

```
AzHostActivityLayout(navController, initiallyExpanded = false) {
    azTheme(activeColor, defaultShape, headerIconShape, translucentBackground)
    azConfig(packButtons, dockingSide, noMenu)
    azAdvanced(helpEnabled, helpList = activeHelpList, onDismissHelp, tutorials)
    if (isRailVisible) ConfigureRailItems(...)
    background(weight = 0) { MainScreen(...) }
    onscreen { AzNavHost(startDestination) { composable(...) } }
}
```

Help is wired through a `HelpViewModel`-backed `StateFlow<Map<String, Any>>` collected as `activeHelpList` and passed to `azAdvanced`. Tutorials are produced by `getTutorials(layers, strings): Map<String, AzTutorial>` and passed to `azAdvanced`. Tutorial start/end uses `LocalAzTutorialController.current`.

### 3.2 The actual problem

GraffitiXR maintains **two parallel help mechanisms**:

1. The `azAdvanced(helpList = ...)` map, used by the help overlay (the documented 8.9 mechanism).
2. A per-item `info = context.getString(strings.help.X)` argument on every `azRailHostItem` / `azRailSubItem` / `azRailItem` call (~50+ sites in `MainActivity.kt:1029-1615`).

The second mechanism is the source of drift. Recent commits `ff0e83ea` ("Fix IllegalArgumentException and update Trace mode tutorial"), `c9e6b79f` ("Fix IllegalArgumentException when opening sub-menus"), and `f38250c5` are symptoms of mismatched IDs between rail items, the `info` argument's expected anchor lookup, and tutorial `AzHighlight.Item(id)` references. The 8.9 changelog explicitly calls out "Resolve identifier collisions, isolation leaks, and dependency …" — that is the upstream fix for the same disease.

### 3.3 Why content audit comes with structural migration

Once `info = ...` is removed and `helpList` becomes the single source of truth, every help string surfaces through the same overlay UI. Inconsistent voice, length, or staleness becomes visible in one place. The audit is cheap to do alongside the structural change because every string moves through the same edit; doing it later means re-reading every site.

## 4. Architecture

### 4.1 Decisions

| Decision | Choice |
|---|---|
| Container | Keep `AzHostActivityLayout`. No structural change. |
| Help source | `helpList` passed to `azAdvanced` is the **only** mechanism. Every `info = ...` argument is removed from rail items. |
| Tutorial registration | Keep `azAdvanced(tutorials = ...)`. No structural change. |
| Tutorial trigger | Keep existing logic at `MainActivity.kt:494-513` (auto-show on mode change when no layers + tutorial unread). Mirror to `settingsViewModel.markTutorialComplete("tut_${mode}")` for back-compat with the existing `completedTutorials` gate. |
| Rail item IDs | Single namespace, single helper: `layerId(layer, tool: String? = null)`. Sanitize non-`[A-Za-z0-9_-]` characters via stable hash fallback. Static IDs use dotted form (`mode.host`, `target.create`, `project.settings`). |
| Color customization | If 8.9 dropped the `color` parameter on `azRailHostItem` / `azRailSubItem`, color customization moves to `azTheme.activeColor` plus per-item `content = Color(...)` for always-visible color. Verified during plan execution by compiler errors. |

### 4.2 Layer ID helper

```kotlin
private fun layerId(layer: Layer, tool: String? = null): String =
    if (tool == null) "layer.${sanitize(layer.id)}"
    else "layer.${sanitize(layer.id)}.tool.$tool"

private fun sanitize(s: String): String =
    if (s.matches(Regex("[A-Za-z0-9_-]+"))) s
    else "h${s.hashCode().toUInt().toString(16)}"
```

Producer and consumer share this function. `allHelpItems`, `ConfigureRailItems`, and tutorial-anchor sites in `TutorialDefinitions.kt` and `GraffitiTutorials.kt` all call it. There is no other way to construct a layer-tool ID in the codebase. ID collisions become structurally impossible.

## 6. Content audit scope

Strings not listed below are presumed acceptable and not reviewed.

Mandatory review candidates (most-likely-stale):

- Mode strings: presumed fresh (Trace was just reworked). Mode tutorial cards: same.
- Design menu: `addImg`, `addDraw`, `addText`, `wall`.
- Tools menu: `flashlight`, `lockTrace`.
- Layer-tool strings (one of: keep, rewrite, mark for follow-up): `size`, `font`, `color`, `kern`, `bold`, `italic`, `outline`, `shadow`, `stencil`, `blend`, `adj`, `invert`, `eraser`, `blur`, `liquify`, `dodge`, `burn`, `iso`, `line`, `balance`.
- Project menu: `newProject`, `saveProject`, `loadProject`, `exportImage`, `appSettings`.

Voice / length rules:

- Second person, present tense, action-first ("Tap to add a new layer." not "This adds a new layer.").
- ≤2 sentences for the always-visible help body (the collapsed card).
- ≤6 sentences for expanded card content where applicable.
- No internal jargon (no "splat", "slam", "fingerprint" without context).
- No references to features that have moved/been removed.

Each reviewed string lands as either:
- `verified accurate` (a commit message tag, no string change), or
- a string update.

## 7. Error handling

### 7.1 Eliminated by design

| Failure | How it dies |
|---|---|
| `IllegalArgumentException` from sub-menu open (commits `c9e6b79f`, `f38250c5`) | Debug-only `RailIntegrityCheck` asserts every registered sub-item's `hostId` matches a registered host's `id`. Mismatches `error()` in debug; impossible to ship. |
| `IllegalArgumentException` from tutorial anchor (commit `ff0e83ea`) | `TutorialAnchorTest` (§8) asserts every `AzHighlight.Item(id)` resolves to a real rail-item ID for a fixed sample layer set. CI catches mismatches. |
| ID collisions between layers | `layerId(layer, tool)` is the only producer; `RailIntegrityCheck` asserts `ids.size == ids.toSet().size`. |
| Help text drift between `info = ...` and `helpList` | `info = ...` is gone. |

### 7.2 Tolerated, made graceful

| Failure | Behavior |
|---|---|
| `helpList` missing entry for a registered rail item | Library default: label-only, no body text. Debug-only `Log.w("RailIntegrity", ...)` lists orphans on first paint. |
| `controller.startTutorial(id)` for an unregistered id | Library default: silent no-op. Defensive `if (id in tutorials)` at call sites. |
| User dismisses tutorial mid-scene | `onSkip` → `controller.markTutorialRead(id)` + `settingsViewModel.markTutorialComplete("tut_${mode}")`. |
| User completes tutorial | `onComplete` → same as skip. |
| Library version mismatch (8.9 dropped something we missed) | Compile-time error during plan execution. Treated as primary signal. |

### 7.3 RailIntegrityCheck

Debug-only, runs once per composition of `ConfigureRailItems`:

1. All rail-item IDs unique. (Failure: `error()`.)
2. Every sub-item's `hostId` matches a registered host. (Failure: `error()`.)
3. Every key in the active `helpList` matches a registered rail-item ID. (Failure: `Log.w`.)
4. Every `AzHighlight.Item(id)` in the registered tutorials matches a registered rail-item ID. (Failure: `Log.w`.)

All four blocks live behind `if (BuildConfig.DEBUG)`.

## 8. Testing

| Test | Asserts |

| `TutorialIdsRegisteredTest` | Every key in `getTutorials(...)` matches an ID reachable by the trigger logic at `MainActivity.kt:494-513`. |

### 8.2 Compile-time verification

- 8.9 API removals surface as compile errors during the version bump.
- If `kotlinOptions.allWarningsAsErrors = true` is not set on the `app` module, set it; deprecation warnings on the bumped library would otherwise hide.

### 8.3 Manual verification (run during plan execution, before declaring done)

1. Fresh install. Launch each editor mode (AR, Overlay, Mockup, Trace). Confirm corresponding tutorial auto-appears.
2. Tap `tool.helpMain`. Tap each rail item in the overlay. Confirm correct help body for each.
3. Open every host item (`mode.host`, `target.host`, `design.host`, `project.host`). Zero `IllegalArgumentException`.
4. Add three layers with deliberately ugly IDs (emoji, spaces, slash). Open each layer's nested rail. Open help overlay. Verify each tool shows correct help.
5. Trigger Trace mode tutorial. Confirm cards render.
6. Toggle `target.scanModeCycle`, `tool.light`, `tool.lockTrace`. State and help text consistent.
7. Five-minute exploratory session covering every rail item. Zero `IllegalArgumentException`.

### 8.4 Deliberately not tested

- Visual snapshot tests (overlay rendering churns between library versions; brittle).
- Localized help strings (out of scope per §2).
- Help-map rebuild performance (`remember(layers)` keying is sufficient until proven otherwise).

## 9. Definition of done

- `app` and all feature modules build clean against AzNavRail 8.9 with no deprecation warnings.
- Four unit tests pass (§8.1).
- Seven manual verification steps pass on a debug build (§8.3).
- Zero `IllegalArgumentException`s in the five-minute exploratory session.
- Mandatory content-audit candidates (§6) are each either tagged `verified accurate` in a commit message or rewritten.

## 10. Rollback

Single-PR rollback is sufficient. The migration is structural (no data migration, no schema change, no on-disk format change). Tutorial-read persistence is library-managed and stable across the version bump (key `az_navrail_read_tutorials`).
